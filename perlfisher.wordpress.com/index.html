Rewriting Perl code for Raku Part V

Last week we started to talk about the pack() and unpack() builtins for Raku and Perl. These aren‚Äôt terribly common built-ins to use, so I thought I‚Äôd take some time to go over these in detail and talk about how I use them and debug files that use them.

As a gentle reminder, OLE::Storage_Lite is a Perl module to read and write a subset of the Microsoft OLE storage format. As part of my effort at the start, I‚Äôve got a ‚Äútranslation‚Äù of the original Perl code pounded out, without much thought to whether it‚Äôll work, or really even compile. It looks like the Perl version, but with most of the {} changed to <> and -> changed to ..

What to test first‚Ä¶ The reading side seems to be the easiest, because I can check object-by-object to see what the data should look like. Replicating that for Raku becomes essentially fixing the bugs I know I‚Äôve introduced on the way.

Testing testing‚Ä¶ is this on?

Before we dive into the Raku code, though, let‚Äôs just set up a quick test in Perl. There really wasn‚Äôt one to begin with, which is a testament to how well-used the module is. I‚Äôve got a ‚Äòtest.xls‚Äô file that I‚Äôve already checked in LibreOffice to make sure it works, so I‚Äôll add a test script that reads the file and checks the root object.

use Test::More;
use OLE::Storage_Lite;

my $root = OLE::Storage_Lite->new( 'sample/test.xls' );
use YAML; die Dump($root);
isa_ok $root, 'OLE::Storage_Lite::PPS::Root';
is $root->No, 0;
is $root->PrevPps, 0xfffffffe;

You might be reading the code and wondering what the heck die() is doing in a test suite. It‚Äôs not because in my current copy it‚Äôs commented out, but it‚Äôs a quick and dirty way to get the data for the Raku version of the file, which looks almost the same.

use Test;
use OLE::Storage_Lite;

my $root = OLE::Storage_Lite.new( 'sample/test.xls' );
die $root.perl;
isa_ok $root, 'OLE::Storage_Lite::PPS::Root';
is $root.No, 0;
is $root.PrevPps, 0xfffffffe;

Notice there‚Äôs hardly any difference overall, just a few minor syntax tweaks. And I don‚Äôt need to use YAML. But I‚Äôve got a Q&D way to run my code, and since my screen looks something like this:

I‚Äôve got most of what I need in my face. This is all a rather plain TMUX setup, running multiple panes so I can see what‚Äôs going on. On the left is vim running in split-screen mode with the Perl and Raku test files open. The rest are shells in the Perl and Raku directories, and some commands to get byte dumps of the files.

I‚Äôve also in my shells set up the following aliases:

alias 5 = "perl -Ilib"
alias 5p = "prove -Ilib"
alias 6 = "perl6 -Ilib"
alias 6p = "prove -e'perl6 -Ilib'"

This way I can run both Perl and Raku test suites with just a few keystrokes, and not have to worry about details such as -I paths. You‚Äôre of course welcome to do things exactly the same, completely different, or even radically better than I am, in which case please let me know.

You might notice the use of the language‚Äôs old name here. I haven‚Äôt changed over to the new binaries yet, but the techniques I‚Äôll talk about here won‚Äôt change.

Keeping it Clean

We now have two scripts that should produce the same output, but probably won‚Äôt, for any number of reasons. I‚Äôve got a whole article‚Äôs worth of things that I had to do to make the new module compile, let alone run. But that‚Äôs for a later issue.

Let‚Äôs start out with this section, which might be familiar to longtime (ha!) readers. 

  $rhInfo->{_FILEH_}->seek(0, 0);
  $rhInfo->{_FILEH_}->read($sWk, 8);
  return undef unless($sWk eq "xD0xCFx11xE0xA1xB1x1AxE1");

This is in Perl, of course. In Raku I‚Äôve chosen to write

  $file.seek( 0, SeekFromBeginning );
  my Str $sWk = $file.read( 8 ).unpack('A8');
  die "Header ID incorrect" if $sWk ne HEADER-ID;

It‚Äôs a bit ungraceful to die() inside a module, but this guarantees that execution stops way before it can cause a hard-to-debug problem down the road. The first change is that I‚Äôve refactored $rhInfo->{_FILEH_} out into its own $file variable so I don‚Äôt have to repeat references to $rhInfo all over the place, like the original.

Next is using the built-in IO::Handle constant ‚ÄòSeekFromBeginning‚Äô instead of the rather anodyne 0 as in Perl. Probably the parent OLE::Storage module looked ahead in the file to determine something before reading in earnest. I‚Äôm keeping it here for no good reason other than it might be nice to separate ‚Äòread‚Äô functionality into a different method.

Diving in

The next line will cause some consternation, so I‚Äôll unpack it slowly. The original author used Hungarian notation for their variable names, so the ‚Äòs‚Äô of $sWk means that it‚Äôs a string type. I‚Äôve adopted this for the Raku code as well, actually enforcing the variable type without additional code.

File handles have both a fancy lines() method that lets you read files line-by-line, and a raw read()method that lets you read raw bytes. If I stopped right here and just looked at the raw bytes, the code would actually fail, and I‚Äôve talked about why in earlier parts. Suffice to say that read()returns a buffer of uninterpreted bytes, not a string that you have to decode later.

Decoding here is the job of the unpack() statement. It acts just like its Perl counterpart, but is experimental. Lucky for me, it implements enough of the Perl builtin that I can use it to read the entire OLE file. 

Now, unlike other builtins (again, keeping in mind it‚Äôs experimental,) it‚Äôs only available as a method call. There is a version of unpack() that works on multiple arguments, but if you try to call it as a builtin, expect:

===SORRY!=== Error while compiling -e
Undeclared routine:
    unpack used at line 1. Did you mean 'pack'?

This may be fixed in your version, feel free to try it and let me know if I should upgrade üôÇ In any case, the last bit you‚Äôre wondering about is the ‚ÄòA8‚Äô business as its argument. I think this isn‚Äôt explained correctly in the documentation, so I‚Äôll explain in my own way.

read() returns a raw string of bytes, without interpretation. If it sees hex 041, it doesn‚Äôt ‚Äúknow‚Äù if you meant the ASCII character ‚ÄòA‚Äô or the number 41, so it doesn‚Äôt interpret the data, it just puts the data into the buffer. It relies on the the Buf(fer)‚Äôs pack() and unpack() methods to assign types to the data.

So finally, unpack( "A8" ) pulls out 8 ‚ÄúASCII‚Äù characters and puts them into $sWk. Now I used scare-quotes there because ASCII is a 7-bit encoding, not 8 bits as many people seem to think. It only encodes from 0x00-0x7f, so anything over that isn‚Äôt legal ASCII.

Which just means that the ‚ÄúA‚Äù of ‚ÄúA8‚Äù doesn‚Äôt truly correspond to ASCII, but it‚Äôs close enough. So, we call unpack( ‚ÄúA8‚Äù ) on the buffer that $file.read( 8 ) returns, and get back a string that we can finally check against our header.

Debugging

But what if the header isn‚Äôt what we expect? Your first instinct might be to say you must‚Äôve screwed up and sent it the wrong file. Luckily it‚Äôs pretty easy to check that, just call $file.slurp.print; That‚Äôll tell you the contents quickly. If it‚Äôs text you‚Äôve probably got the wrong file ‚Äì OLE files do contain text but it‚Äôs usually zip‚Äôed or in UCS-2.

Let‚Äôs assume though that it‚Äôs an actual binary file, and a real spreadsheet that Excel (or LibreOffice in my case) can read. Since the headers don‚Äôt match, it must be a different version of OLE that our code isn‚Äôt ready to handle.

That means we need to know what the first 8 bytes of the file actually are. We‚Äôve got a bunch of tools at our disposal, but what I want to introduce is hexdump(1) (don‚Äôt worry about the (1), force of habit.) Run this command on the file:

hexdump -C sample-file.xls | head -1

This should generate something like this:

00000000  d0 c9 11 a0 af b1 13 d1  00 00 00 00 00 00 00 00  |................|

(original bytes changed to protect the innocent file) The numbers on the left (‚Äò00000000‚Äô) tell us how far we are into the file (in hex), the next two groups of 8 are the hex values of the individual bytes of the file, and the dots between ‚Äò|..|‚Äô are where any printable characters would appear, if there were any.

So now we know what the first 8 bytes of this file look like, and we can add (without much muss or fuss) some checks to our original file, and come up with this:

$file.seek( 0, SeekFromBeginning );
my Str $sWk = $file.read( 8 ).unpack('A8');
die "Unknown OLE header!" if $sWk eq "xd0xc9x11xa0xafxb1x13xd1";
die "Header ID incorrect" if $sWk ne HEADER-ID;

This check isn‚Äôt in my source, so don‚Äôt go looking for it. As far as I know there aren‚Äôt any other OLE header strings than what I check for, but then I‚Äôm trying to get away without reading the spec. My blood pressure doesn‚Äôt need that.

Getting at the details

Of course, binary packed formats contain more stuff than just ASCII strings. OLE was originally written in the days of 16-bit CPUs, so it‚Äôs got other ways to pack in data. Let‚Äôs look at a fragment of the file format: (not from the spec, this is just my interpretation)

0000: 0xD0 0xCF 0x11 0xE0 0xA1 0xB1 0x1A 0xE1 # header
0008: 0x00 0x09           # size of large block of data (in power-of-2)
000a: 0x00 0x06           # size of small block of data
000c: 0x00 0x00 0x00 0x03 # Number of BDB blocks
0010: 0xff 0xff 0xff 0xfe # Starting block

So, this is the first 20 (0x0010+4) bytes of an OLE header block. You may have already caught on to the fact that there are at least 3 sizes of data here. The first 8 bytes on line 0000 is the header data we talked about ad nauseam.

Next, the header says that a ‚Äúlarge‚Äù block of data is 2**9 bytes long, and a ‚Äúsmall‚Äù block of data is 2**6 bytes long, this time in pairs of bytes. Finally we‚Äôve got the number of BDB blocks (whatever those are, probably Berkeley DB) and the starting block‚Äôs index number, all in 4-byte chunks.

This means we need to read 2 2-byte chunks and 2 4-byte chunks into memory. This time though, we have to read them as numbers. Once again, unpack() comes to the rescue. Last time we used the ‚ÄòA‚Äô character, this time we‚Äôll do something just a little bit different.

Let‚Äôs read the documentation for unpack() to see what we can use. Halfway down the page we come to a table which gives us the letter abbreviations for each type of data we can read, and what it is in terms of where it is in memory.

For now, replace the term ‚Äòelement‚Äô with ‚Äòbyte‚Äô while you‚Äôre reading the documentation. We need to read (0x00, 0x09) as a 2-byte integer, so let‚Äôs look for ‚Äútwo elements‚Äù on the right-hand side. ‚ÄúExtracts two elements and returns them as a single unsigned integer‚Äù seems to be what we need.

So it looks like the letter we need to use is ‚ÄúS‚Äù, and since we only want to read one at a time, that‚Äôs all we need. But the original Perl source uses ‚Äúv‚Äù, so that‚Äôs what I‚Äôll use as well.

  $iWk = _getInfoFromFile($rhInfo->{_FILEH_}, 0x1E, 2, "v");
  return undef unless(defined($iWk));
  $rhInfo->{_BIG_BLOCK_SIZE} = 2 ** $iWk;

But as you can see, the Perl source creates a wrapper around the pack() method, much to my annoyance. I‚Äôd prefer to simply write this:

$iWk = $file.read( 2 ).unpack( "v" );
%hInfo<_BIG_BLOCK_SIZE> = 2**$iWk;

but to keep things looking as similar to the original Perl code as I can, my code looks like 

  my Int $iWk = self._getInfoFromFile( $file, 0x1E, 2, "v" );
  die "Big block size missing" unless defined( $iWk );
  %hInfo<_BIG_BLOCK_SIZE> = 2 ** $iWk;

which is just one line longer, and that‚Äôs because of the safety check. Of course, pack() and unpack()can take more than one format character at time. In Perl, there‚Äôs yet another mini-language (like regex, and what used to be called the format statement) for these builtins, and that‚Äôs not quite done yet.

But you can still take the entire header we‚Äôve collected so far, and write it into a single unpack()statement like so:

my ( $header, $large-size, $small-size, $num-bdbs, $start-block ) =
  $file.read( 20 ).unpack( "A8 vv VV" );

This format is of course much more compact and much easier to read. In all probability once I get done with the main module I‚Äôll convert everything over to this style and the code will become much, much quieter. Binary protocols, especially those for moisture evaporators, tend to have lots of code that looks like:

my $rev = $file.read(2).unpack("v")
if $rev == 0x01 {
  $r2 = $file.read(2).unpack("v");
} else {
  $d2 = $file.read(4).unpack("V");
}

where the next bytes you read depend upon the version of the protocol. Even though I‚Äôve just been rattling off code based on the Perl version, I don‚Äôt know what the protocol may do at any given point. So it makes sense to read just one int or long ahead while developing.

I could read a version number as ‚ÄúV‚Äù because they started out using ‚Äúv1‚Äù, ‚Äúv2‚Äù and so on up to ‚Äúv42792643522‚Äù. But then 30 lines and 2 revs later they may have changed from ‚ÄúV‚Äù to ‚Äúvcc‚Äù because they wanted to support ‚Äúv2.1.0‚Äù style.

And if that header were something like ‚ÄúA8 V CC* V vv‚Äù I have to go back and break up the format string and statement at the very least. If I go term-by-term I just have to find the version number and add an if-then statement just below.

Now that you‚Äôve got a fairly good grounding in unpack(), I think it‚Äôs time for break. Next time we‚Äôll cover writing our file back out, the most fun part of the operation.

Rewriting Perl Code for Raku IV: A New Hope

Back in Part III of our series on Raku programming, we talked about some of the basics of OO programming. This time we‚Äôll talk about another aspect of OO programming. Perl objects can be made from any kind of reference, although the most common is a hash. I think Raku objects can do the same, but in this article we‚Äôll just talk about hash-style Perl objects.

Raku objects let you superclass and subclass them, instantiate them, run methods on them, and store data in them. In previous articles we‚Äôve talked about all but storing data. It‚Äôs time to remedy that, and talk about attributes.

Instance attributes

We used unit class OLE::Storage_Lite; to declare our class, and method save( $x, $y ) { ... } to create methods. Or in our case rewrite existing functions into methods. Now, we focus our attention on some of the variables that should really be instance attributes, and why.

Let‚Äôs get to know which variables behave like attributes, and which don‚Äôt. This will change how we write our Raku code, but hopefully for the better. We‚Äôll start from the outside in, and look at the API. There are a few ‚Äútest‚Äù scripts that use the module, and this fragment is pretty common.

use OLE::Storage_Lite;
my $oOl = OLE::Storage_Lite->new('test.xls');
my $oPps = $oOl->getPpsTree(1);
die( "test.xls must be a OLE file") unless($oPps);

The author creates an object ($oOl) from an existing file, then fetches a tree of ‚ÄúPps‚Äù objects, whatever they are. So, one OLE::Storage_Lite object equals one file. This gives me my first instance variable, the filename.

sub new($$) {
  my($sClass, $sFile) = @_;
  my $oThis = {
    _FILE => $sFile,
  };
  bless $oThis;
  return $oThis;
}

Above is how they wrote it in Perl, and below is how we‚Äôd write it (exactly as specified) in Raku:

has $._FILE;

multi method new( $sFile ) {
  self.new( _FILE => $sFile );
}

Later on, we can call my $file = OLE:Storage_Lite.new( 'test.xls' ); just like we did in Perl. We wouldn‚Äôt even need the new method if we had users call my $file = OLE::Storage_Lite.new( _FILE => 'text.xls' );. This gives users the option of calling the API in the old Perl fashion or the new Raku fashion without additional work on our part.

Strict Raku-style

There‚Äôs a problem lurking here, though. The constructor Raku provides us lets us call my $file = OLE::Storage_Lite.new(); without specifying a value for $._FILE. If you know Perl‚Äôs Moose module, though, the ‚Äòhas‚Äô there just might look familiar.

And for good reason. A lot of the ideas from Moose migrated into Raku during its design, and the attributes were one of those. Moose lets you do a lot of things with attributes, and so does Raku. One of those is you can add ‚Äúadverbs‚Äù to them. Let‚Äôs do that now.

has $._FILE is required;

Calling OLE::Storage_Lite.new() now fails, because you‚Äôre not passing in the _FILE argument. That solves one problem. Actually, it solves two, come to think of it. In the original Perl code, you could call OLE:Storage_Lite->new() too, and it wouldn‚Äôt complain. Now we‚Äôve fixed that, with one new term.

Progressive Typing

No, we‚Äôre not talking about some new editor like Comma (the link does work, despite the certificate problem.) Our code would run just fine, as-is. Users could call our .new() API, Raku would make sure the filename existed, and we could go on with translating.

But there‚Äôs something more we can take advantage of here, and that is the fact that any Raku object (and anything we can instantiate is an object) is a type as well. We haven‚Äôt mentioned that because we really couldn‚Äôt use that information until now.

The original Perl code is littered with clues to types, hidden in the variable names. When we wrote our own API call, the Perl code called the file name $sNm. The ‚Äòs‚Äô tells the Perl compiler nothing, but it tells us that $sNm is a String type. Perl may not have true types, but Raku does. Let‚Äôs fix our attribute with that in mind.

has Str $._FILE is required;

We knew all along that $._FILE is a string of some sort, but telling Raku that lets it allocate space more efficiently. Making sure it‚Äôs a required attribute lets anyone that calls new() know if they forget an argument. We could go a little farther with this, but locking down attributes will help in the long run, when we start dealing with the pack and unpack built-ins.

Packing It All In

We‚Äôre now getting to the heart of the module. There‚Äôs a lot of mechanics above us, allocating objects and doing math and checking types, and not much below us. The class‚Äô entire purpose is to read and write OLE-formatted files. We‚Äôll talk more about the boilerplate, but here‚Äôs the real meat of the file.

Let‚Äôs start with what should be simple, reading in data. Just like in Perl, we open a file and get back a ‚Äúfile handle‚Äù (assuming the file exists, of course.) By default, calling my $fh = open $._FILE;gives us a read-only file handle. The file handle itself has a bunch of attributes associated with it, but the important one right now is its encoding.

Namely, the fact that it has none. An OLE file is essentially a miniature filesystem (probably based on FAT) packed onto disk, complete with a root directory, subdirectories and files. File have names encoded in UCS-2, but the rest is entirely dependent upon what the application requires.

The upshot of which is that we can‚Äôt read the format with something simple like my @lines = $fh.lines; which would read line after line into the @lines array. Instead we‚Äôll use calls like read()and write() that return byte-oriented buffers.

Buffering‚Ä¶

All OLE files start off with the header ‚ÄúxD0xCFx11xE0xA1xB1x1AxE1‚Äù, so we should probably start there. That‚Äôs important twice in the code, in fact. First, when we‚Äôre reading off disk, we can check it against what we‚Äôve just read to make sure this file is OLE, and not, say, a JSON file. Later on, when we‚Äôre saving out an OLE file, we can write it as the header string.

constant HEADER-ID = "xD0xCFx11xE0xA1xB1x1AxE1";

I‚Äôll make it a constant as well, so when I revisit this code in a month I don‚Äôt have to go looking in specs for ‚Äò0xd0 0xcf‚Äô to remember what this is. Reading is straight-forward too. It needs just a byte count.

my Buf $header = $fh.read( 8 );

Something important to notice here is the type, ‚ÄòBuf‚Äô. If our file was in Markdown, or JSON we could get away with just writing my @lines = $fh.lines; like I tried earlier. But these are raw bytes, hindered by no interpretation. Let‚Äôs see what happens when we compare these bytes to our HEADER-ID.

t/01-internals.t ............ Cannot use a Buf as a string, but you called the Stringy method on it
  in method _getHeaderInfo at /home/jgoff/GitHub/drforr/raku-OLE-Storage_Lite/lib/OLE/Storage_Lite.pm6 (OLE::Storage_Lite) line 169
  in block <unit> at t/01-internals.t line 42

Another brick in the wall

Ka-blam. But‚Ä¶ hold the phone here a minute, I just said $header eq HEADER-ID, I didn‚Äôt write anything like ‚ÄòStringy‚Äô! There‚Äôs no ‚ÄòStringy‚Äô in the source‚Ä¶ oh. HEADER-ID is a string, so Raku is being helpful. I‚Äôm trying to use string comparison (‚Äòeq‚Äô) between something that‚Äôs not a Str ( $header ) and something that is (HEADER-ID).

Pull up the Stringy documentation, and look for the Type graph. Midway down you‚Äôll see ‚ÄòBuf‚Äô and ‚ÄòStr‚Äô, as of this writing Buf is on the left, and Str is popular so it‚Äôs in the middle.

Trace the inheritance paths from Buf and Str upwards, and you‚Äôll see they pass Buf -> Blob -> Stringy and Str -> Stringy, and stop. What the error message therefore is saying is this, anthropomorphized:

You wanted to convert Buf to Str, and didn‚Äôt care how you did it. So I looked. First, on the Buf type. No .Str method there, at least without arguments. No good. So I looked in its parent, Blob. Nothing doing there. Then I looked at Stringy, and couldn‚Äôt find anything else.

There‚Äôs nothing above me, nothing below. So I‚Äôll let you know I looked for a conversion method in a bunch of places, stopped at Stringy, and couldn‚Äôt go any farther. Sorry.

Raku

You‚Äôre probably wondering how to get out of this quandary. Reading the Blob documentation closely, you might think that the encode method is the way out of our present jam. If you look closer, though, there‚Äôs a spanner in the works. ‚ÄúxD0‚Äù is the byte 0xd0, so if you try to decode to ASCII, you run into the problem that ASCII only covers 0x00-07xf, everything outside of that is undefined.

Packing for vacation

If you‚Äôve kept up with things, you might surmise by now that the key to our quandary lies in the pack and unpack builtins. Specifically unpack(), because we‚Äôre trying to ‚Äúdecode‚Äù a buffer into something suitable for Raku.

Unless you‚Äôve done things like network programming or security, the pack and unpack builtins are going to be unfamiliar territory. The closest analogue of pack() is the builtin sprintf().

Both of these builtins take a format string telling the compiler how to arrange its arguments. Both of them take a mixture of string and integer arguments afterwards. But while sprintf() takes the arguments and treats its output as a UTF-8 encoded string, pack() takes the same arguments and treats its output as a raw buffer of bytes.

And now you can see one way out of our little predicament. If we could just find the right invocation, pack() would be able to take our string ‚Äúxd0xcf‚Ä¶‚Äù and turn it into a Buf object. Then we could compare the buffer we got by reading 8 bytes to the buffer we expected.

So instead of cluttering up the main code, let‚Äôs write a quick test.

use experimental :pack;
constant HEADER-ID = "xD0xCFx11xE0xA1xB1x1AxE1";

use Test;
my $fh = open "test.xls";
my Buf $buf = $fh.read( 8 );

is $buf, pack( "A8", HEADER-ID ); # Pack 8 ASCII characters

Testing‚Ä¶testing‚Ä¶

Let‚Äôs take it from the top. We tell Raku to use the ‚Äúexperimental‚Äù pack() builtin, and declare the header we want to check against. Then we tell Raku we want to use the Test module, and open a new Microsoft Excel test file.

Last, we read a chunk of 8 bytes from the file into a buffer, and check to see that the 8 bytes matches the header we expect to see. Now, how did we get that weird ‚ÄòA8‚Äô string in there? I thought pack() looked more like sprintf()?

Well, it does, to an extent. I/O routines like sscanf() and sprintf() can do all sorts of things to your strings and numbers on the way in and out, think for example what ‚Äò%-2.10f‚Äô means in a format specifier, for instance. You can follow along with the unpack() documentation if you like.

pack(), by contrast, just takes 8, 16, or 32-bit chunks of your input, and places them into a buffer. The ‚ÄúA‚Äù in ‚ÄúA8‚Äù says that it wants to convert an ASCII-sized chunk of your input (‚Äúxd0‚Äù in our case) into a byte in the buffer, so our Buf now looks like ( 0xd0 ).

I could just as well have said ‚ÄúAAAAAAAA‚Äù in order to translate all 8 characters of the buffer, but I think it‚Äôs a little tidier to use the ‚Äòrepeat‚Äô option, and say ‚ÄúA8‚Äù in order to convert just 8 characters (yes, yes, I know, they‚Äôre glyphs, but let‚Äôs not confuse matters.)

I could write ‚ÄúA*‚Äù just as well, but ‚ÄúA8‚Äù makes sure that 8 and only 8 (the number that thou shalt count to‚Ä¶) characters get converted. I doubt that the header in an OLE file will change, but it‚Äôs a nice bit of forward planning.

Rewriting Perl Code for Raku III: The Sorceror

Last week, we started testing, learned how to create proper Raku classes, and the basics of functions. This time we‚Äôll take a closer look at functions, arguments, and make some decisions about the API. And maybe while writing this I‚Äôll argue myself out of a decision. It‚Äôs happened before.

It‚Äôs at odds with some of the principles I laid down at the start, which states that I should have minimal changes in the API from Perl to Raku. In Perl 5, you would create the ‚Äúfilesystem root‚Äù object like so:

my $root = OLE::Storage_Lite::PPS::Root.new(
  [ 0, 0, 0, 25, 1, 100 ],
  [ 0, 0, 0, 25, 1, 100 ],
  [ $workbook, $page_1, $sheet_1 ]
);

In fact, all I did was copy and change two characters, specifically the Perl ‚Äò->‚Äô to the Raku ‚Äò.‚Äô operator. Clean, and very simple. And I think what I‚Äôll do is actually just change the code back to using the Perl reference, at least in the API. Dereferencing it will be just a few lines, and I‚Äôll have to change it in the tests as well, but I think the pain will be worthwhile.

This way I don‚Äôt have to field questions like ‚ÄúWhy did you end up potentially breaking old code?‚Äù during talks. See, speaking at conferences about your code really¬†can¬†be a useful motivator!

I‚Äôd like a formal argument, please

So, I think I‚Äôve settled on Perl-style formal references, at least for the current iteration. There are actually better ways to do this, but I‚Äôll leave that for the proper Raku version. For right now, quick-n-dirty is the name of the game.

Moving on, we see an important method in the original Perl code, saving an object to disk.

sub save($$;$$) {
  my($oThis, $sFile, $bNoAs, $rhInfo) = @_;
  #0.Initial Setting for saving
  $rhInfo = {} unless($rhInfo);
  # ..
}

As I‚Äôve mentioned before, OLE::Storage_Lite has been around for a long, long time. And it‚Äôs obvious here. Function prototypes (not signatures, which are a different kettle of fish) and the use of ‚Äò$oThis‚Äô instead of the more conventional ‚Äò$self‚Äô.

Being prototypical

Prototypes were originally meant as a way to save you from having to write checks in your code. Theoretically, if your function was called sub save($$) and you tried to call it with save($fh) you would get an error, because the ‚Äò$$‚Äô means the subroutine took two arguments, and you gave it just one.

But it also predated objects (yes, Virginia, objects in Perl haven‚Äôt been around all that long.) and they could have unforeseen side effects. So they were a fad for a while, but quickly faded out of existence.

These days they‚Äôre a reason for a more experienced Perl hacker to take the junior aside and explain quietly why we don‚Äôt use those anymore, and point them to some modern references, like Modern Perl (not an affiliate link, yet.)

Let‚Äôs at least partially convert that to Raku, like so:

method save($sFile, $bNoAs, $rhInfo) {
  #0.Initial Setting for saving
  $rhInfo = {} unless($rhInfo);
  # ..
}

The ‚Äò$oThis‚Äô means that this is a method call, so instead of writing sub save( $oThis, ... ) we can rewrite it to a method and gain ‚Äòself‚Äô instead of the arbitrary variable ‚Äò$oThis‚Äô. Of course we do have to do a search-and-replace on ‚Äò$oThis‚Äô with ‚Äòself‚Äô, but that‚Äôs relatively simple. More complex is what to do with the ‚Äò;‚Äô in the original prototype.

Having options

It‚Äôs worth pointing out that OLE::Storage_Lite is taken at least in part from another (larger) module, OLE::Storage. This means that the internal code is redundant in a few places. Raku would let us rewrite what we have as:

method save($sFile, $bNoAs, $rhInfo = {}) {
  #0.Initial Setting for saving
  # ..
}

making $rhInfo an optional variable with a default value. Now, this is a pretty common pattern for a recursive method, so I did a bit of digging. Namely I grep‚Äôed for ‚Äòsave‚Äô in the original (all-in-one) Storage_Lite.pm module, and found no recursive calls to it.

Debugging both sides now

This is also where the test suite I wrote earlier comes in handy, as it actually exercises the ‚Äòsave‚Äô method. So I added a quick debugging message warn "Saving $rhInfo"; to my local copy of the code, and ran the test suite. Seeing just one ‚ÄòSaving ‚Ä¶‚Äô message in my test output convinced me it wasn‚Äôt recursive. So now the code just looks like:

method save($sFile, $bNoAs) {
  #0.Initial Setting for saving
  my %hInfo;
  # ..
}

Also, since $rhInfo is created in this method, there‚Äôs no reason to leave it as a reference. So the initial ‚Äòr‚Äô goes away, and we have left just ‚Äò%hInfo‚Äô. It may get passed in to other methods, but Raku lets us pass hashes and arrays as ordinary variable types, so I‚Äôll take advantage of that.

To be fair, leaving it as a reference would have saved me a bit of typing, but I‚Äôd already kind of decided that at least internally I‚Äôd try to use Raku types and calling conventions, and that left me with the choice of how to pass variables around.

Having options

Finally, there‚Äôs the question of what to do with the semicolon. Remember at the start, the function prototype was ‚Äò($$;$$)‚Äô which meant $oThis and $sFile were before the semicolon, and $bData and $rhInfo were after. I can now reveal that ‚Äò;‚Äô in a Perl prototype means that whatever appears afterward is optional.

True to Raku‚Äôs nature, I can account for this in at least two ways. One way would be to decide that $bData is always there and just has a default value, probably 0. That would look like method save( $sFile, $bData = 0 ). But the documentation puts $bData in square brackets, indicating that it‚Äôs optional.

Raku has an alternate syntax to indicate if a variable is optional, which looks like method save( $sFile, $bData? ). I think this method is better than the alternative syntax because it states clearly that $bData is optional. Both methods work, I just happen to like the ‚Äò?‚Äô modifier.

Waiting for Huffman

Moving on, we have this wonderful line of code:

$rhInfo->{_BIG_BLOCK_SIZE}  = 2**
              (($rhInfo->{_BIG_BLOCK_SIZE})?
                  _adjust2($rhInfo->{_BIG_BLOCK_SIZE})  : 9);

When I was translating this initially, I was in something of a drone mindset, not truly thinking about what I was doing. I‚Äôd copied the $rhInfo variable into the method signature and just kept on writing. I ended up with a statement that I eventually shortened quite a bit.

$rhInfo.<_BIG_BLOCK_SIZE> = 2**
  ( $rhInfo.<_BIG_BLOCK_SIZE> ??
    _adjust2( $rhInfo.<_BIG_BLOCK_SIZE> ) !!
                                        9 );

The ‚Äò.‚Äô after $rhInfo indicates we‚Äôre dealing with a reference, and the <..> notation is now how barewords look inside hashes. The old {_BIG_BLOCK_SIZE} is still there, but it‚Äôs pronounced {‚Äò_BIG_BLOCK_SIZE‚Äô}. A lot of people use the {‚Äò..‚Äô} in Perl already so it‚Äôs not a big change, and it actually simplifies the backend enormously.

Also, at the start Larry and Damian pulled statistics on Perl code from CPAN and other repositories. They were looking for operator frequencies, among other things. Frequently used operators like qw() and -> got even shorter in Raku.

Others, like the ternary operator, weren‚Äôt so lucky. It got longer, and stretched to ‚Äò?? .. !!‚Äô. So this is one place where the code will look a little funky. Maybe one day I‚Äôll write a slang to fix it, but back to work.

Trimming the verge

Earlier I mentioned that this module was trimmed down from a much larger full OLE reader/writer. This was the first place that became evident. Since $rhInfo is now called %hInfo and initialized inside the method, this statement deserves to be looked at a little closer.

my %hInfo;
%hInfo<_BIG_BLOCK_SIZE> = 2**
  ( %hInfo<_BIG_BLOCK_SIZE> ??
    _adjust2( %hInfo<_BIG_BLOCK_SIZE> ) !!
                                        9 );

After replacing $rhInfo with %hInfo this is what I got. But since %hInfo is defined just above, the test %hInfo<_BIG_BLOCK_SIZE> will never be true, so this entire block can be reduced to:

my %hInfo = _BIG_BLOCK_SIZE => 2**9;

While I‚Äôm here I‚Äôll delete _adjust2(). No code pathway uses it, so out it goes. I‚Äôll restore it if I have to, but right now I want the test scripts to pass, and that‚Äôs it. I‚Äôve got the original source, and a map from Perl to Raku, and that‚Äôs all I need.

Culling yaks from the herd

Where there‚Äôs smoke there‚Äôs fire, so I stop what I‚Äôm doing and grep out every ‚Äòsub X‚Äô call in the source, putting it in a scratch monkey. Then I go through the source (which I have below the new Raku source, deleting lines as I go) and look for methods that aren‚Äôt used, like adjust2(). I delete each of these methods with extreme prejudice, because each line of code I don‚Äôt see is one I don‚Äôt have to translate.

Checkpoint in git, and now it‚Äôs time for a lunch break. Afterwards, I‚Äôm getting into the save()method, and see what looks like a new yak to shave. Or a package to translate, to be precise.

  if(ref($sFile) eq 'SCALAR') {
    require IO::Scalar;
    my $oIo = new IO::Scalar $sFile, O_WRONLY;
    $rhInfo->{_FILEH_} = $oIo;
    # ...
}

In both Raku and Perl, you can create a single method called new( $sFile ) that treats $sFile as either a filename (scalar), file content (scalar reference) or file handle (scalar object.) In Perl, if we wanted to handle filenames, file contents, or file handles differently, we‚Äôd have to switch like this, or have different method names.

In Raku, we can handle this differently. In fact I can write the code to save() to a filename, and add save() to a filehandle later with no modifications needed. Above, I briefly touched on the fact that you can write more than one new() method, as long as the two method signatures were distinct.

multi method save( IO::Handle $fh ) {...}

Raku will let you write two methods called save(), as long as it can tell which one to call at runtime. So, I can call $root.save( '/tmp/test.xlsx' ); or $root.save( $out_filehandle ); and Raku will ‚Äúdispatch‚Äù it to the right save() method automatically.

We call it ‚Äòmultiple dispatch‚Äô for just that reason, dispatching a function call to multiple versions of a method. And this means that I can write the first save( Str $filename ) method without worrying about the other methods. I don‚Äôt have to add a new if-then branch to the existing code, or modify save() in any way.

I can just write my save() method and ignore the other IO:: types. Also, if someone gets my code later and wants to add a save() method that saves to something I know nothing about, they can write their new save() method without interfering with mine.

Making things functional

To keep things simple for me writing the code, and me having to read the code weeks, months or years later, I want as close to a 1:1 relation between Perl 5 and Raku as I can. Under Construction

And there we are. Now, there‚Äôs quite a bit to take in, so I‚Äôll take things slow. The first thing you‚Äôll notice is the keyword ‚Äòmulti‚Äô. In Perl 5, you get to hand-roll your own constructors, so you can make them any way you like. In this case, the author chose to write new($raTime1st, $raTime2nd, $raChild), which is pretty common.

Raku gives me a default ‚Äònew‚Äô method, so I only need to hand-roll constructors when I want. Since I want to keep as close as reasonable to the original API, I‚Äôll write a constructor that takes 3 arguments too. In my case I chose to simplify things just a bit here.

I‚Äôve found over several years of writing Raku code that I rarely use references. In Perl 5 they were pretty much the only way to pass arrays or hashes into a function, because of its propensity to ‚Äúflatten‚Äù arguments.

In Raku, you can still use the Perl 5 style, but formal argument lists are the way to go in my opinion. If you need to pass both an array and a hash to a Raku function, go for it. I encourage that in my tutorial courses, and recommend it to help break students out of their Perl 5 mindset.

This is not to say that there‚Äôs anything wrong with Perl 5‚Äôs argument list, in fact they‚Äôve taken some ideas from Raku for formal argument lists, and I encourage that. Cross-pollination of ideas should be encouraged, it‚Äôs how both languages grow and add new features.

Templates II: Electric Boogaloo

Last time on this adventure writing the Template Toolkit language in Raku, we‚Äôd just created a small test suite that encompasses some of the problems we‚Äôre going to encounter. # ... similar lines above this
is-deeply the-tree( 'xx[% name %]x' ),
    [ 'a', 'a', Directive.new( :content( 'name' ) ), 'a', ];
# ... and similar lines below this.

The list here is what we‚Äôre going to return to render(), and I‚Äôd love to make that as simple as it can be without being too simple. Let‚Äôs focus for the moment just on one bit of the test suite here, the array I‚Äôm getting back.

If these elements were all strings, then all render() would have to do is join the strings together, simples!

method render( Str $text ) returns Str {
  my @terms = # magic to turn text into array of terms
  @terms.join: '';
}

Let‚Äôs create the ‚ÄòDirective‚Äô class and see what happens, though.

class Directive { has $.content }

my @terms = 'a', 'a', Directive.new( :content( 'name' ) ), 'a';
say @terms.join: '';
# aaDirective<94444485232315>a

Whoops, that‚Äôs not what we want. Not bad exactly, but not what we want, either. Well, not to fear. Remember that in Template Toolkit, directives will always return a string. It may be an emptystring, but they‚Äôll always return some kind of string.

As a side note, this may not always be true ‚Äì some directives will even tell the renderer to stop parsing entirely. But it‚Äôs a pretty solid starting assumption. For instance, we could say that encountering the STOP directive just makes all future directives return ‚Äù.

Of course, I‚Äôm harping on the term ‚Äòstring‚Äô for a reason. Internally, everything is an object, and every object has a method that returns a readable value. Our Directive class didn‚Äôt specify one, so we get the default that returns ‚Äò$name<$address>‚Äô.

So, let‚Äôs supply our own method.

class Directive { has $.content; method Str { $.content } }

my @terms = 'a', 'a', Directive.new( :content( 'name' ) ), 'a';
say @terms.join: ', ';
# a, a, name, a

There. If we supply a .Str method we can make Directives do what we want. INCLUDE directives would open the file, slurp the contents and return them. Argument directives would take their argument name, look up the value, and return that. Or, more likely, would have a context object passed that does the lookup for them.

Next in this series on writing your own template language using Raku, you should be able to define your own Template Toolkit directives and have them return the pre-processed text. A Regex amuse-bouche

Before continuing with the Template series, I thought I‚Äôd talk briefly about an interesting (well, at least to me) solution to a little problem. System and user libraries (the kind that end in .so or .a, not Perl libraries) have a section at the top that maps a function name (‚Äòload_user‚Äô or whatever) to an offset into the library, say, 0x193a.

This arrangement worked fine for many years for C, Algol, FORTRAN and most other languages out there. But then along came languages that upset the apple cart, like C++ and Smalltalk, where a programmer could write two ‚Äòload_user‚Äô functions, call ‚Äòload_user(1234)‚Äô or ‚Äòload_user(‚ÄúSmith, John‚Äù)‚Äô and expect the linker to load the right version of ‚Äòload_user.‚Äô

The problem here is that the library, the linker and all of the other programs in the tool chain expect there to only be one function called ‚Äòload_user‚Äô in any given library.

Those of us that do Perl 5 and Raku programming don‚Äôt have to worry about this, but if you ever want to link to a C++ library, you probably should know at least a bit about ‚Äúname mangling.‚Äù

For a while, utilities like ‚ÄòCFront‚Äô for the Macintosh (which the author actually filed bug reports on) were used to ‚Äúrename‚Äù functions like ‚Äòload_user(int)‚Äô and ‚Äòload_user(char*)‚Äô to ‚Äòi_load_user‚Äô and ‚Äòcs_load_user‚Äô before being added to the library, and other tools to do the reverse.

Has Your Mother Sold Her Mangle?

Eventually things settled down, and this process of changing names to fit into the library was ‚Äúbaked in‚Äù to the tool chains. Not consistently, of course, couldn‚Äôt have that. But conventions arose and even today Wikipedia lists at least 12 different ways to ‚Äúmangle‚Äù ‚Äòvoid h(void)‚Äô into the existing library formats.

We‚Äôll just look at the first one, ‚Äò_Z1hv‚Äô. The ‚Äò_Z‚Äô can be safely ignored, its purpose there is mainly to tell the linker something ‚Äúspecial‚Äù is going on. ‚Äò1h‚Äô is the function name, and ‚Äòv‚Äô is its first (and only) parameter. Suppose, then, that you were tasked with writing a tool that undid this name mangling.

Your first cut at extracting something useful might look something like

'_Z9load_useri' ~~ m{ ^ '_Z' \d+ (\w+) (.) $ };

Assuming $mangle-me has ‚Äò_Z9load_useri‚Äô in it (The mangled version of ‚Äòvoid load_user(int)‚Äô) the regex engine goes through a bunch of simple steps.

  • Read and ignore ‚Äò_Z‚Äô
  • Read and ignore ‚Äò9‚Äô
  • Capture ‚Äòload_user‚Äô into $0
  • Capture ‚Äòi‚Äô into $1
  • There is no fifth thing.

But the person that wrote this library is playing silly buggers with someone (obviously us in this case) and there‚Äôs also a ‚Äò_Z9load_userss‚Äô which comes out of the other end of the mangle looking like ‚Äòvoid load_user(char*, char*)‚Äô, loading a user with first and last names.

Now we‚Äôre in a bit of a quandary. Run the same expression and see what happens:

'_Z9load_userss' ~~ m{ ^ '_Z' \d+ (\w+) (.) $ };

Sure enough, $1 is ‚Äòs‚Äô, just as we wanted it, but what about $0? It‚Äôs now ‚Äòload_users‚Äô, which‚Ä¶ y‚Äôknow, looks too legit to quit. But we must. And now we‚Äôre faced with the quandary. Do we make the first parameter an optional capture? ‚Äòm{ ‚Ä¶ (.)? (.) $ }‚Äô like so?

No, that would capture the ‚Äòr‚Äô of ‚Äò_Z9load_users‚Äô. There must be something else in the name that we‚Äôre overlooking, some clue‚Ä¶ Aha! ‚Äòload_user‚Äô has 9 characters, and look just before it, we‚Äôve got the number 9! Surely that tells us the number of characters in the function name! (and thankfully it actually does.)

Regexes 201

Now, how can we use this to our advantage? First things first, let‚Äôs get rid of some dead weight. We don‚Äôt care (for the moment) about parameters, so let‚Äôs just match the name and number of characters. And because we‚Äôre getting all serious up in here, let‚Äôs create a quick test.

use Test;
'_Z9load_user' ~~ m{ ^ '_Z' (\d+) (\w+) };
is $0, '9';
is $1, 'load_user';

Run the test script, see if it passes, I‚Äôm sure you know the drill. Go ahead and copy that, I‚Äôll wait. Okay, the tests pass, so it‚Äôs time to play. I usually am working in a library that‚Äôs in git, so I‚Äôm usually on the ‚Äúedit, run tests, git reset, edit‚Ä¶‚Äù treadmill by this point.

So‚Ä¶ How do we make use of this number? Well, let‚Äôs pull up the Regexes page over at docs.raku.organd look around. Back in Perl 5 there used to be this feature ‚Äòm{ a{5} }x‚Äô that matched just 5 copies of whatever it was in front of, that might be a good place to start looking.

That‚Äôs now morphed into ‚Äòm{ a ** 5 }‚Äô. Great, so let‚Äôs replace 5 with $0 and go for it.

'_Z9load_user' ~~ m{ ^ '_Z' (\d+) (\w ** $0) };

‚ÄúQuantifier quantifies nothing‚Ä¶‚Äù That‚Äôs weird. $0 is right there, staring me in the face. Maybe I just got the syntax wrong somehow?

'_Z9load_user' ~~ m{ ^ '_Z' (\d+) (\w ** 9) };

Nope, that works. What‚Äôs going on here? $0 is defined‚Ä¶ Wait, it‚Äôs a variable inside a regex, that used to require the ‚Äòe‚Äô modifier, didn‚Äôt it? Or something like that‚Ä¶ <read the manpage, scratch head‚Ä¶ nothing there> Hm. Are we at a dead end?

Kick it up a notch

No, we just need to remember about how string interpolation works. In Raku, ‚ÄúHello, {$name}!‚Äù is a perfectly fine way to interpolate variables into your expression, and it works because no matter where it is, {} signals a code block. Let‚Äôs try that, surround $0 with braces.

'_Z9load_user' ~~ m{ ^ '_Z' (\d+) (\w ** {$0}) };

Weird. This time the test failed with ‚Äù instead of ‚Äòload_user‚Äô. Maybe $0 really isn‚Äôt defined? Now that it‚Äôs just regular Raku code, let‚Äôs check.

'_Z9load_user' ~~ m{ ^ '_Z' (\d+) (\w ** {warn "Got '$0'"; $0}) };

‚ÄúUse of Nil in string context.‚Äù So it‚Äôs really empty. Now, we have to really do some reading. Looking at the section on general quantifiers says ‚Äúonly basic literal syntax for the right-hand side of the quantifier [what we want to play with] is supported,‚Äù so it looks like we‚Äôre at a dead end.

But things like ‚Äò{$0}‚Äô do work, so we can use variables. That means that my problem isn‚Äôt that the variable is being ignored, it‚Äôs just not being populated when I need it. Let‚Äôs look at the section on Capture numbers to see when they get populated.

Aha, you need to ‚Äúpublish‚Äù the capture using ‚Äò{}‚Äô right after it. Let‚Äôs see if that works‚Ä¶

'_Z9load_user' ~~ m{ ^ '_Z' (\d+) {} (\w ** {warn "Got '$0'"; $0}) };

Nope, something else is going on. And the next block down tells us the final solution ‚Äì ‚Äò:my‚Äô. This lets us create a variable inside the scope of the regular expression and use it as well, so let‚Äôs do just that.

'_Z9load_user' ~~ m{ ^ '_Z'
                     :my $length;          # Put $length in the proper scope
                     (\d+) {$length = +$0} # Capture the length
                     (\w ** {$length})     # And extract that many chars.
                   };

And reformat things just a wee bit so we‚Äôve got some room to work with. Now the test actually runs, and reads only as many characters of the function name as needs be.

And just one more thing‚Ä¶

It‚Äôs not just function names that follow this pattern, it‚Äôs also namespaces, and any special types that the function might use as parameters, so let‚Äôs package this up into something more useful.

my regexp pascalish-string {
  :my $length;
  (\d+) {$length = +$0}
  (\w ** {$length})
};
'_Z9load_user' ~~ m{ ^ '_Z' <pascalish-string> };
is $/<pascalish-string>[0], 9;
is $/<pascalish-string>[1], 'load_user';

Pascal implementations were done back when RAM was at more of a premium, and stored a string like ‚Äòload_user‚Äô as ‚Äò\x{09}load_user‚Äô so the compiler knew how many bytes were available immediately rather than having to guess. It was limiting, but this was on computers like the early Macs (we‚Äôre talking pre-OS X, for that matter pre-System 7, for those of you that remember that far back.)

So we can use this <pascalish-string> regular expression anywhere we want to match one of our counted terms. Because we‚Äôre using ‚Äòmy‚Äô inside a regular expression nested inside another regular expression inside a burrito wrapped in an enigma, there are no scoping troubles.

There are probably other ways of doing this, and I would love to see them. If you do come up with a better way to solve this, let me know in the comments and I‚Äôll work your solution into an upcoming article.

Let‚Äôs leave that decision for now, and go on.

Retrograde is all the rage

Let‚Äôs apply some basic retrograde logic to what we‚Äôve got here, given what we know of Raku tools. In order to get the string ‚Äò<h1>Hello, Jeff!</h1>‚Äô from ‚Äò<h1>Hello, [% name %]!</h1>‚Äô, we need a lot of mechanics at work.

At first glance, it seems pretty obvious that ‚Äò[% name %]‚Äô is a substitution marker, so let‚Äôs just do a quick regexp like this:

$text ~~ s:g{ '[%' (\w+) '%]' } = %args{$0};

That should replace every marker in the text with something from an %arguments hash that render() supplies to us. End of column, end of story. But not so fast, if all Template Toolkit supplied to us was the ability to substitute values for keys, then ‚Ä¶ there‚Äôs really no need for the module. And in fact, if you look at the docs, it can do many more things for us.

For example, ‚Äò[% INCLUDE %]‚Äô lets us include other template files in our own, ‚Äò[% IF %]‚Äô .. ‚Äò[% END %]‚Äô lets us do things conditionally, and a whole host of other ‚Äúdirectives‚Äù are available. But you‚Äôll see here the one thing they have in common is they all start with ‚Äò[%‚Äô and end with ‚Äò%]‚Äô.

Hold the phone

That isn‚Äôt entirely true, and in fact there‚Äôs going to be another article in the series about that. But it‚Äôs a good starting point. We may not know much about what the language itself looks like, but I can tell you that tags are balanced, not nested, and every ‚Äò[%‚Äô opening tag has a ‚Äò%]‚Äô tag that closes it.

I‚Äôll also point out that directives ( ‚Äò[% foo %]‚Äô ) can occur one after another without any intervening white space, and may not occur at all. So already some special cases are starting to creep in.

In fact, let‚Äôs put this in as a separate test file entirely. So separate that we‚Äôre going to put it in a nested directory, in fact. Let‚Äôs open t/parser/01-basic.t and add this set of tests:

Now just HOLD THE PHONE here‚Ä¶ we‚Äôre testing directives for Template Toolkit, not binary numbers, and whatever that other column is! Well, that‚Äôs true. We want to test text and directives, and make sure that we can get back text when we want it, and directives when we want them.

At first blush you might think it‚Äôs just enough to make sure that ‚Äò<h1> Hello,‚Äô is parsed as text, and that ‚Äò[% name %]‚Äô is parsed as a directive, and just leave it at that. But those of you that have worked with regular expressions for a while might wonder how ‚Äò[% name %][% other %]‚Äô gets parsed‚Ä¶ does it end at the first ‚Äò%]‚Äô, or continue on to the next one?

And what about text mixed with directives? Leading? Trailing text? Wow, a lot of combinations. In fact, if you wanted to be thorough, it wouldn‚Äôt hurt to cover all possible combinations of text and directives up to‚Ä¶ say, 4 in a row.

Let‚Äôs call text ‚ÄòT‚Äô, and directives ‚ÄòD‚Äô. I‚Äôve got 4 slots, and only two choices for each. Filling the first slot gives me ‚ÄòT_ _ _‚Äô and ‚ÄòD_ _ _‚Äô, for two choices. I can fill the next slot with ‚ÄòT T _ _‚Äô, ‚ÄòT D _ _‚Äô, ‚ÄòD T _ _‚Äô, and ‚ÄòD D _ _‚Äô, and I think you can see where we‚Äôre going with this. 

In fact, replace T with 0 and D with 1, and you‚Äôve got the binary numbers from 0000 to 1111. So, let‚Äôs take advantage of this fact, and do some clever editing in our editor of choice:

Logic Programming in Raku

This is a small example of conference-driven development. I‚Äôm sitting in the board room at TPCiP ‚Äì TCP in Pittsburgh surrounded by people doing both Perl 5 and Raku programming, and decided to look again at Picat, working on some simple examples. I was thinking that I might be able to translate some of the simpler backtracking examples from Picat to Raku and here‚Äôs a simple example.

First the Picat code:

fib(0,F) => F=1.
fib(1,F) => F=1.
fib(N,F),N>1 => fib(N-1,F1),fib(N-2,F2),F=F1+F2.

Now here‚Äôs my equivalent Raku code:

multi fib( 0, $F is rw ) { $F = 1 }
multi fib( 1, $F is rw ) { $F = 1 }
multi fib( $N is rw where * > 1, $F is rw ) {
  my ( $F1, $F2 ) = 0, 0;
  my $N1 = $N - 1;
  my $N2 = $N - 2;
  fib( $N1, $F1 ) && fib( $N2, $F2 ) && $F = $F1 + $F2
}

The Raku version is slightly larger because I need to declare some variables that Picat would ordinarily declare for me ($F1, $F2). There may be a way to work around declaring ($N1, $N2), but otherwise the two versions are identical.

How does it work?

You‚Äôve probably guessed based on the inputs that N is index of the Fibonacci number, and F is the Fibonacci number itself. Picat doesn‚Äôt require you to declare variables, so you could ask it for the 7th Fibonacci number by calling fib(7,F) and looking at F.

my $Fib = 0;
fib(7,$Fib);
say $Fib     # 21

Or you could do the above in Raku, letting the code populate $Fib for you. This code relies on the fact that Raku lets you dispatch not just on signatures, not just on argument types, but on values. Look at the base case above:

multi fib( 0, $F is rw ) { ... }

fib(‚Ä¶) is the function signature, and this function will get called whenever the first argument is 0, like so: fib(0, $Fib). This happens even if ‚Äòmulti fib( $N, $F )‚Äô is the one doing the calling, everything gets run through the same dispatcher each time.

So fib(2, $Fib) calls fib(1, $Fib) which calls ‚Äòmulti fib( 1, $F )‚Äô and gives us a base case, for example. This lets the higher-order functions call our base cases, and still get the right value.

What are we missing?

Well, the Picat code can do something the Raku code can‚Äôt, at least for the moment, and this is what I want to spend some time working on. In Picat, I can call ‚Äòfib(6,F)‚Äô and F will be 13 when the code is done. This works in Raku too.

But Picat will also let you call ‚Äòfib(N,21)‚Äô and N will be 7 when the calculation is finished. Take some time to let that settle. Yes, you can run the calculation both forward and backward. Give N a value, and F will be the Nth Fibonacci number. Give F a value, and it will tell you what N is.

In fact, Picat will go one step further. If you don‚Äôt specify a value for either parameter but just specify variables, like ‚Äòfib(N,F)‚Äô, then it will generate all the Fibonacci numbers and their indexes until you tell it to stop.

This is because of the backtracking engine that it uses, which I want to see if I can mimick. ‚ÄòF=F1+F2‚Äô doesn‚Äôt mean ‚ÄúAssign the sum of F1 and F2 to F‚Äù. Instead, ‚ÄúIf any values are missing, find values that satisfy the equation, and keep generating them until you run out of possibilities.‚Äù

That‚Äôs a bit of a mouthful, so let‚Äôs look at just F1. Supposing F=8 and F2=5, the backtracking engine would search all values of F1, and return just the matching value of 3. Now of course, it can‚Äôt search all values, because that means you‚Äôd be waiting forever, so there are pruning algorithms at work here.

But the same logic can work with any combination of arguments, so if both F1 and F2 were missing, then the backtracking engine would run through all possible combinations of values (pruned appropriately) until it found a combination that would work.

In this case, since in our example F=8, it would return a bunch of combinations, starting with (F1=1, F2=7), (F1=2, F2=6) and so on. But why, then, you ask, does it only return (F1=3, F2=5)? That‚Äôs because each value F1 also has to satisfy fib(N1,F1), which means that F1 has to be a Fibonacci number, as does F2.

Breakdown

This is the part where Raku breaks down a little bit. But what I think I might be able to do is use a trick I used a while ago, relying on the fact that operators are just functions, and they dispatch just like other functions. So I should be able to start out with something crude like:

my $F = Operator.new( :lhs(3) );
my Value ($F1, $F2);
$F = $F1 + $F2;

This way both $F1 and $F2 are bound to backtracking Values, they return a Operator, and the Operator is part of the backtracking engine. This way once the Operator engine determines the range of possible combinations of $F1 and $F2 that add to 3, it can assign them concurrently to @F1.value and @F2.value.

Smooth Operators

The Operator and Value classes, along with their overloaded operators, would look something like this:

class Operator {
  has ( $.lhs, $.rhs ); }
class PlusOperator is Operator { }
class AssignmentOperator is Operator {
  method make-combinations() {...} }
class Value {
  has @.value }
multi infix:<=>( Operator $lhs, Operator $rhs ) {
  AssignmentOperator.new( :$lhs, :$rhs );
}
multi infix:<+>( Value $lhs, Value $rhs ) {
  PlusOperator.new( :$lhs, :$rhs );
}

This is purely a sketch that I haven‚Äôt tried out at all. My idea here is that once you‚Äôve executed ‚Äò$F = $F1 + $F2‚Äô, $F will be an AssignmentOperator instance. You should be able to call $F.make-combinations() that will solve the equation ‚Äò3 = $F1 + $F2‚Äô for all (constrained) values of $F1 and $F2.

That would populate @F1.values and @F2.values with (1,2) and (2,1) respectively. I‚Äôm about to play my first game of Azul, so I‚Äôll leave the article here. The next article will hopefully implement this so you can see this all working. It won‚Äôt quite be a true backtracking engine, but it‚Äôs a start.

Quantum Tunneling

Introducing the new Perl Fisher site

Before I get on to the meat of the article, welcome to the new home of The Perl Fisher. I intend to cover both Perl 5 and Raku programming here, but it‚Äôll be mostly Raku content because that‚Äôs the language I find the most fun. Please excuse the dust, I‚Äôm still very much settling into the new home, and the overall look of the site is bound to change while I play with the new toys available to me.

Defeating Thanos with Raku

Don‚Äôt worry, no spoilers here. We‚Äôre just going to talk about a little-known feature of Raku, the quantum-tunneling variable type. If you‚Äôve worked with Raku for any length of time, you‚Äôve probably seen or written a class declaration that looks something like below.

class Point2D {
  has Real $.x;
  has Real $.y;
}

While the word ‚Äòhas‚Äô does the real work, second-sigil syndrome strikes as well, in the shape of the ‚Äò.‚Äô between the scalar sigil ‚Äò$‚Äô and the variable name. Here it‚Äôs syntactical sugar for being an attribute name, but we can enlarge that ‚Äò.‚Äô to a ‚Äò*‚Äô and open up a world of possibilities.

When we add the ‚Äò*‚Äô sigil to a variable name, we turn that variable into one that can quantum tunnel between scopes and solve problems that you probably used to do with a global variable. You can read more about dynamic variables and how they differ from ordinary globals at The_*_twigilat docs.raku.org.

Testing, testing

I‚Äôm working on a project to try to augment the Raku grammar debugger with an emulator. The tools we have on CPAN and modules.perl6.org respectively are wonderful, but they‚Äôre limited because Raku compiles grammar rules down to single methods, which is wonderful for speed, but makes it almost impossible to look into.

The grammar I‚Äôm writing isn‚Äôt important at the moment, but the testing part is. Below is a sample subtest that I‚Äôm writing for each term of a grammar that‚Äôs probably going to have ~50 terms by the time I‚Äôm done.

subtest 'binary-number', {
  subtest 'failing', {
    ok fails( '0b', 'binary-number' );
    ok fails( '3g', 'binary-number' );
  };

  is build-ast( '0101', 'binary-number' ), 5;
};

This tests the ‚Äòbinary-number‚Äô rule to see if it properly fails on ‚Äò0b‚Äô and ‚Äò3g‚Äô. ‚Äò0b‚Äô fails because it‚Äôs the prefix of a binary number, and ‚Äò3g‚Äô because neither 3 nor ‚Äòg‚Äô are binary digits. It also makes certain that ‚Äò0101‚Äô gets translated into the decimal number 5. All important when testing a grammar that parses ‚Ä¶ well, itself eventually. Oroborous redux, as it were.

Dry up, will you‚Ä¶

The test is simple, and straightforward. ‚Äò0b‚Äô should fail, ‚Äò0101‚Äô should be built into a node of an abstract syntax tree. But it‚Äôs got some flaws. It talks too much. See how ‚Äòbinary-number‚Äô repeats itself? If I want to copy that, rename it to ‚Äòhex-number‚Äô and add a few changes, I have to copy the block, rename all the incidences of ‚Äòbinary-number‚Äô to ‚Äòhex-number‚Äô and then fix the existing tests.

Thus I run the risk of forgetting to update the name ‚Äòbinary-number‚Äô. And there‚Äôs an even greater bugaboo there. If I don‚Äôt, the test won‚Äôt fail. Because the subtest doesn‚Äôt know that it‚Äôs supposed to be testing the ‚Äòbinary-number‚Äô rule. There are a bunch of ways to solve this problem, of course, but for this post we‚Äôre going to use Ant-Man(tm).

Entering the Quantum Realm

I don‚Äôt want to do too much work here, I just want to get rid of the duplicate ‚Äòbinary-number‚Äô entries. So, let‚Äôs take a look at what fails() does.

sub fails( Str $sample, Str $rule-name ) returns Bool {
  !?( $g.parse( $sample, :rule( $rule-name ) );
}

The ‚Äò!?(‚Ä¶)‚Äô casts $g.parse(‚Ä¶) to a Boolean and negates it, so if $g can‚Äôt parse the statement, it returns True. So, first let‚Äôs make $rule-name optional. 

sub fails( Str $sample, Str $rule-name? ) returns Bool {
  !?( $g.parse( $sample, :rule( $rule-name ) );
}

Opening the wormhole

Now, we‚Äôre going to summon Ant-Man(tm). Remember earlier I mentioned that quantum variables use a wormhole? Well, we‚Äôre going to open one end of the wormhole right here in our fails() function, just like this.

sub fails( Str $sample, Str $rule-name? ) returns Bool {
  !?( $g.parse( $sample, :rule( $*ANT-MAN // $rule-name ) );
}

Rerun our tests, and ‚Ä¶ wait, they should fail, we haven‚Äôt declared $*ANT-MAN anywhere! Well, just like in quantum physics, $*ANT-MAN doesn‚Äôt have enough energy to tunnel over the quantum barrier because we haven‚Äôt defined him yet. 

So let‚Äôs do that, but remember that $*ANT-MAN is a quantum variable, so he can tunnel through the quantum barrier of a function scope. In fact, he can tunnel through any number of them. So, let‚Äôs define a new version of subtest() that looks and acts like the old one first before we go boldly where no Raku programmer has gone before.

sub Subtest( Str $rule-name, Block $test-code ) {
  subtest $rule-name, $test-code;
}

We should be able now to replace the outer subtest() block with our new Subtest() block, and it should act just as it used to.

Subtest 'binary-number', {
  subtest 'failing', {
    ok fail( '0b', 'binary-number' );
    ...
  };
  ...
};

Tunneling through

Our test suite still works, and the output still is what we expect. Now, let‚Äôs give $*ANT-MANenough energy to tunnel through the quantum barrier by defining him as the subtest name we want:

sub Subtest( Str $rule-name Block $test-code ) {
  my $*RULE-NAME = $rule-name;
  subtest $rule-name, $test-code;
}

And now run our test suite. Which‚Ä¶ doesn‚Äôt change. Come to think of it, we don‚Äôt want it to change. If it did change, we‚Äôd have to go through and change all of our test suites, which would be bad. So, putting things together, this code works just fine.

sub fails( Str $sample, Str $rule-name? ) returns Bool {
  !?( $g.parse( $sample, :rule( $*ANT-MAN // $rule-name ) );
}
sub Subtest( Str $rule-name Block $test-code ) {
  my $*RULE-NAME = $rule-name;
  subtest $rule-name, $test-code;
}

Subtest 'binary-number', {
  subtest 'failing', {
    ok fails( '0b', 'binary-number' );
  };
};

Notice by the way that $*ANT-MAN has tunneled through not one but two function signatures to get to where he is. And to prove it, finally, delete the inside ‚Äòbinary-number‚Äô.

Subtest 'binary-number', {
  subtest 'failing', {
    ok fails( '0b' );
  };
};

So ‚Äòbinary-number‚Äô gets passed along to $*ANT-MAN who jumps into the quantum realm, tunnels through the outer and inner pairs of braces, and finally lands in fails() where he passes the value on to the :rule() declaration. Whew, that‚Äôs a lot of work.

Oh, snap.

(sorry, couldn‚Äôt resist.) If you‚Äôre like me, and I know I am, you‚Äôve probably come across a few cases where this technique would come in handy. Especially if you‚Äôre dealing with legacy code. Sometimes you need to add just one little flag to a function and set that flag in a top-level handler on one page of a website.

The catch is that between the lower level and the top level there‚Äôs a chain of 8 function calls where you have to add that as a parameter. Wouldn‚Äôt it be nice if there was a workaround? Well, in Raku there is.
